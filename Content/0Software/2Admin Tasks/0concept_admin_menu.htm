Detection Statistics

This module is responsible for the pre-process of the raw power spectra data, in an attempt to maximize the SNR for a particular type, or class, of disturbance (See Preliminary Process Flow Diagram). The objective is to isolate the energy that comes from the disturbance type the system is set up to detect, from all the other energy that can excite the sensor.

The Detection Statistic Module applies sophisticated signal process techniques to the instantaneous power spectra, to compute an optimal Detection Statistic. The Detection Statistic Module separates disturbance energy into five types of sub-classes:

  • Analysis band
  • Signal intensity relative to the noise floor (strong or weak)
  • Temporal dynamics (persistent or impulsive)
  • Spatial extent (localized or non-localized)
  • Spectral shape (low frequency modeled, low frequency un-modeled or non-low frequency)

The output produced by Helios is based on a Kalman filter, which averages the power spectra to assist in the detection response from subtle low intensity disturbances. This utilises a two dimensional array of data, with distance on one axis, and time shown on the other axis. For more information, refer to your training materials.

The Detection Statistics Module consumes the most compute resources of all the modules. For this reason, the number of simultaneous Active Detection Statistic templates should be kept to a minimum. Typically, a persistent stream and an impulsive stream are computed in parallel. For disturbances with different spectral content, a new Detection Statistic template can be created that only changes the Analysis Band. As this process is at the tail end of the Processing flow, it will not add very much compute load to the system.

Indicators...

This module is used to create an Indicator cleaning mask. Indicator Cleaning Masks take raw indicators and attempt to remove false positive and false negative values that are not associated with the desired disturbance.

The raw indicators are cleaned for distance, and then time. Event sizes refer to distances in metres and Event durations refer to times in seconds. It is important to remember the cleaning order, as it can explain why certain cleaning behaviour is observed. The Indicators (Cleaning) dialog is shown below. Streams...

The output from the Streams Definition Module forms the input to the Alarm Reporting module. Streams consist of a Detection Statistic template and an Indicator Mask (or Indicator Template). In addition, the Detection Threshold used to convert the Detection Statistic into Raw Indicators is defined in the Streams dialog. This is to allow a single Indicator Mask to be re-used with multiple Detection Thresholds. A detection statistic template and Indicator Mask MUST exist before a Stream can be defined. You have to select an optimal Detection Threshold to create the raw indicators, as well as select an input Detection Statistic template and Indicator Mask. If there is no Indicator Mask, refer to Indicator Properties. Below is the Streams Dialog, refer to User Template Properties (Streams) to see a description of the information that goes in each field. Refer to Detection Statistics User Templatesfor instructions about how to create, edit and delete the templates in this dialog.

Alarm Reports...

The Alarm Reporting Module determines if a valid alarm should be generated. This happens when cleaned indicators meet the criteria that have been defined, in order, from the:

  • Basic Tracking
  • Alarm filters
  • Enabled, high-level rules.

Rather than use the cleaned indicators from a single Stream, the cleaned indicators from a number of Streams can be combined together with Boolean logic and then sent to the Alarm Report decision logic.

If more than one stream of indicators is required, they are combined; a Boolean AND operation is used for this. If more than one stream of indicators is disallowed, they are first combined with a Boolean AND operation, and then removed from the required indicator streams.

This is because, all of the required streams MUST BE present, and none of the disallowed streams can be present. For an example, refer to your training materials.
